# ACE EDSL Design Document

## Overview

`ace_edsl` is an Embedded Domain-Specific Language (EDSL) for writing FHE (Fully Homomorphic Encryption) programs in Python. It uses operator overloading to trace Python functions and generate AIR (Abstract Intermediate Representation), which is then lowered through a multi-stage compilation pipeline to produce C code targeting the ANT FHE runtime.

The key design principle: **AIR is generated by executing Python code**, not by parsing or analyzing it. When you write `a + b` inside a `@ckks_kernel` function, the `+` operator emits an AIR node directly.

---

## Architecture

```
Python source (@kernel decorated function)
         │
         ▼
   AST Preprocessor         ← handles loops, if/else, while (BaseDSL infrastructure)
         │
         ▼
     AceEDSL._func_air()    ← executes the function to trace AIR
         │ (operator overloading via AIRValue)
         ▼
     AIR GlobScope          ← in-memory AIR graph (C++ object via ace_bindings)
         │
         ▼  AcePipeline / Pipeline
    ┌────┴──────────────────────────────────────────┐
    │  tensor2vector   nn::core → nn::vector        │
    │  vector2sihe     nn::vector → fhe::sihe       │
    │  sihe2ckks       fhe::sihe → fhe::ckks        │
    │    └── noise budget analysis                  │
    │    └── bootstrap insertion                    │
    │  [ckks_extended_rewrite]  (optional Python)   │
    │  ckks_driver     scale management             │
    │  poly_driver     fhe::ckks → fhe::poly        │
    │  poly2c          fhe::poly → C code           │
    └───────────────────────────────────────────────┘
         │
         ▼
      C source code  (for ANT rtlib / SEAL backend)
```

## Core Components

### 1. `AceEDSL` (`edsl/edsl.py`)

The central DSL class. Inherits from `BaseDSL` (from `base_dsl/`) but **does not use MLIR**. Only the AST preprocessing infrastructure from `BaseDSL` is used.

Key responsibilities:
- Manage the singleton DSL instance (`AceEDSL._get_dsl()`)
- Preprocess function ASTs at decoration time (transform loops/conditionals)
- Execute kernel functions to trace AIR (`_func_air`, `generate_air`)
- Track the current domain and AIR module

**Important**: `BaseDSL` has MLIR dependencies. `AceEDSL` uses a `StubCompilerProvider` to satisfy `BaseDSL.__init__` without triggering MLIR compilation.

### 2. `AIRValue` (`edsl/core/air_value.py`)

A Python wrapper around an AIR node that overloads Python operators to emit AIR operations:

```python
a + b    →  container.new_add(a.value, b.value)
a * b    →  container.new_mul(a.value, b.value)
a[i]    →  container.new_load(a.value, i.value)
```

**Flat IR Mode** (enabled by default): Each operation result is stored to a named temporary (`new_stid`) and loaded back on demand (`new_ldid`). This creates SSA-style flat IR and prevents exponential blowup in the IR dump when operations are deeply nested.

Domain dispatch: `AIRValue` knows its domain (e.g., `fhe::ckks`) and calls domain-specific container methods when available (e.g., `new_ckks_add`).

### 3. Domain Kernel Decorators (`edsl/domain_kernels.py`)

Each decorator (`@ckks_kernel`, `@nn_kernel`, `@sihe_kernel`, etc.) sets a `_py_domain` attribute on the function and wraps it to call `AceEDSL._kernel_helper` at call time.

**Design**: The DSL singleton is looked up at **call time**, not decoration time. This allows test isolation: clearing the singleton cache gives a fresh instance for each test.

Available domains:

| Decorator | Domain string | Typical use |
|---|---|---|
| `@tensor_kernel` / `@kernel` | `air::core` | Generic tensor ops |
| `@nn_kernel` | `nn::core` | Neural network ops |
| `@vector_kernel` | `nn::vector` | Vectorized SIMD-style ops |
| `@sihe_kernel` | `fhe::sihe` | Scheme-independent FHE |
| `@ckks_kernel` | `fhe::ckks` | CKKS-specific operations |
| `@poly_kernel` | `fhe::poly` | Low-level polynomial ops |

### 4. Domain Operations (`edsl/core/`)

Per-domain operation modules providing standalone functions:

- `ckks_ops.py` — CKKS ops: `ckks_add`, `ckks_mul`, `ckks_rescale`, `ckks_relin`, `ckks_rotate`, `ckks_bootstrap`, `bootstrap_coeffs_to_slots`, `bootstrap_eval_mod`, `bootstrap_slots_to_coeffs`, `ckks_conjugate`, `ckks_mul_mono`, `ckks_raise_mod`, `ckks_mod_switch`
- `tensor_ops.py` — Tensor ops: `tensor_add`, `tensor_mul`, etc.
- `vector_ops.py` — Vector ops
- `sihe_ops.py` — SIHE ops
- `poly_ops.py` — Polynomial ops
- `compute_ops.py` / `memory_ops.py` — Lower-level ops

These functions call `container.new_*` methods from `ace_bindings`.

### 5. Compilation Pipeline (`edsl/pipeline.py`)

Two API levels:

**`AcePipeline`** — programmatic API:
```python
pipeline = AcePipeline(glob_scope)
pipeline.configure_fhe(scaling_factor_bits=56)
result = pipeline.run()  # Returns PipelineResult
```

**`Pipeline`** — fluent API (matches acepy style):
```python
result = (Pipeline("my_model")
    .load_onnx("model.onnx")
    .configure_fhe(scaling_factor_bits=56)
    .set_skip_ops(["fhe::ckks::bootstrap"])
    .set_python_lowering(my_lowering)
    .run(target=PipelineTarget.C))
```

Pipeline stages (in order):

| Stage | Transform | C++ pass name |
|---|---|---|
| tensor2vector | `nn::core` → `nn::vector` | `"tensor2vector"` |
| vector2sihe | `nn::vector` → `fhe::sihe` | `"vector2sihe"` |
| sihe2ckks | `fhe::sihe` → `fhe::ckks` (+ bootstrap insertion) | `"sihe2ckks"` |
| [python lowering] | User-defined Python pass | callback |
| [ckks_extended_rewrite] | Lower staged-bootstrap/conjugate/mul_mono to primitives | Python pass |
| ckks_driver | Scale management, rescale insertion | C++ |
| poly_driver | `fhe::ckks` → `fhe::poly` | C++ |
| poly2c | `fhe::poly` → C source | C++ |

**Skip ops**: Operations in `skip_ops` are preserved by C++ passes and can be handled by a Python lowering function registered via `set_python_lowering`.

**`enable_poly=False`**: Stops after CKKS level and emits CKKS-level C (useful for debugging scale management without going all the way to poly).

### 6. Bootstrap Decomposition (`edsl/core/bootstrap_decomposition.py`, `bootstrap_math.py`)

Implements the full algorithmic decomposition of the three CKKS bootstrap stages into CKKS primitives:

- `CoeffsToSlots`: Baby-step/giant-step rotation approach
- `EvalMod`: Paterson-Stockmeyer Chebyshev polynomial approximation + double-angle
- `SlotsToCoeffs`: Inverse of CoeffsToSlots

`bootstrap_math.py` provides:
- Pure-Python PS (Paterson-Stockmeyer) helpers
- CKKS-specific Chebyshev constants (`UNIFORM_HW_192`, degree=54, K=32, R=3)

This decomposition path is activated by:
- `ACE_BOOTSTRAP_STAGE_PRIMITIVE_LOWERING=1` (env var), or
- `Pipeline.set_ckks_extended_op_rewrite(True)`

### 7. CKKS Extended Ops Rewrite Pass (`edsl/passes/ckks_extended_ops_rewrite.py`)

A Python-level pass that walks the AIR graph and rewrites extended CKKS ops (`raise_mod`, `conjugate`, `mul_mono`) into CKKS primitives (`add`/`sub`/`rotate`). Activated by env var `ACE_CKKS_PRIMITIVE_REWRITE=1`.

### 8. Lowering Registry (`edsl/lowering_registry.py`)

Allows registering Python lowering functions for specific operations:

```python
from ace_edsl.edsl.lowering_registry import register_lowering

@register_lowering("fhe::ckks::bootstrap")
def lower_bootstrap(node, container):
    # custom lowering
    ...
```

Operations with registered lowerings can be skipped by C++ passes (via `skip_ops`) and lowered by the Python callback.

## Data Flow: End to End

```
1. User writes:
       @ckks_kernel
       def my_fn(ct: Ciphertext) -> Ciphertext:
           return ct + ct

2. Decoration time:
   - _py_domain = "fhe::ckks" set on my_fn
   - AST preprocessor transforms loops/conditionals

3. Call time: my_fn(some_ciphertext)
   - AceEDSL._kernel_helper called
   - New AIR Container + GlobScope created
   - AIRValue wrappers created for each argument
   - my_fn body executed: ct + ct → AIRValue.__add__ → container.new_add()
   - AIR graph lives in C++ GlobScope

4. AcePipeline.run():
   - tensor2vector / vector2sihe / sihe2ckks run as C++ passes on GlobScope
   - Optional Python lowering callback
   - ckks_driver: scale analysis, rescale insertion
   - poly_driver: polynomial lowering
   - poly2c: emit C source

5. Output: C source file targeting ANT rtlib (libant) or SEAL backend
```

## FHE Parameters (`FHEConfig`)

| Parameter | Default | Meaning |
|---|---|---|
| `poly_degree` | 0 (auto) | RLWE polynomial ring degree (e.g. 65536) |
| `mul_level` | 0 (auto) | Multiplication depth budget |
| `security_level` | 0 (auto) | Security level in bits |
| `scaling_factor_bits` | 56 | CKKS scale factor (precision) |
| `first_prime_bits` | 60 | Size of q0 (first modulus) |
| `hamming_weight` | 192 | Secret key Hamming weight |
| `enable_poly` | True | Whether to lower below CKKS level |

## Key Design Decisions

**No MLIR**: Unlike the base class `BaseDSL`, `ace_edsl` generates AIR directly. MLIR infrastructure is present in `base_dsl` but entirely unused.

**Operator overloading as IR builder**: Python's operator protocol (`__add__`, `__mul__`, etc.) calls C++ AIR builder methods. This is simpler than writing a separate AST visitor.

**Flat IR mode**: Without flattening, nested expressions like `((a+b)*(c+d)) * ((e+f)*(g+h))` produce an exponentially large tree in the AIR dump. Flat mode inserts store/load pairs to break the tree into a linear SSA sequence.

**Domain hierarchy**: The IR has a strict layered domain structure. Passes only lower one level at a time; skipping levels is not supported. This ensures each pass sees a well-typed IR.

**Staged bootstrap**: The three bootstrap stages (CoeffsToSlots / EvalMod / SlotsToCoeffs) can be expressed as first-class IR nodes or decomposed to primitives. The decomposition path allows running bootstrap without the ANT-specific runtime bootstrap precomputation.
